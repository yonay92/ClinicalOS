## Purpose

This document provides an independent architectural assessment of the ClinicalOS specification. It evaluates the design's strengths, identifies risks, flags contradictions, and provides recommendations before any code is written.

---

## 1. Overall Assessment

**Rating: Strong Foundation — Ready to Build with Mitigations**

The ClinicalOS specification is unusually thorough for a pre-build document set. The multi-tenant security model is correctly designed. The event-driven architecture is sound. The AI integration model (suggest, approve, execute) is the right pattern for clinical software. The sprint plan is sequenced correctly.

However, several implementation gaps must be resolved before building begins. These are documented in GAP_ANALYSIS.md. The most critical gap is the Business Rule Engine's undefined JSONB execution schema (GAP-BL-01). Everything else is manageable.

## 2. Architecture Strengths

### 2.1 Multi-Tenant Isolation Design

The `company_id` + `site_id` dual-isolation model is architecturally correct for clinical SaaS. The use of PostgreSQL RLS with helper SQL functions (`current_company_id()`, `has_permission()`) is the right approach — it centralizes authorization logic at the database layer, making it impossible to bypass from the application layer.

**Assessment: Excellent.** This is how enterprise multi-tenant SaaS should be built.

### 2.2 AI as Proposer, Never Executor

The principle "AI suggests, humans approve, Business Rules execute" is the correct model for regulated clinical software. The pipeline `Trigger → AI Analysis → User Review → Business Rules → DB Update → Audit Trail` is well-defined and consistent across all agents.

Storing AI outputs in `ai_responses` with `requires_review = true` and `approved_by = null` creates a natural pending queue that the UI can surface. This pattern is correct.

**Assessment: Excellent.**

### 2.3 Event-Driven Automation via Business Rules

Using configurable Business Rules (stored in `business_rules` table with JSONB conditions/actions) to drive automation is architecturally superior to hardcoded workflows. It allows clinical sites to customize their workflows without code changes, which is critical for enterprise SaaS.

**Assessment: Correct design — execution model needs to be defined (GAP-BL-01).**

### 2.4 Centralized File Architecture

All files flow through the `files` table, with modules linking via `file_links`. This prevents duplication and enables the Enterprise Document Center to serve as a true single source of truth. The private-bucket-only + signed URL approach is correct for HIPAA-adjacent clinical data.

**Assessment: Excellent.**

### 2.5 Complete Audit Trail

The `audit_logs` table structure (`who, what, when, which company, which site, which record, old_value, new_value`) satisfies ICH E6(R3) GCP traceability requirements. The immutability requirement (no UPDATE/DELETE policies on audit_logs) is correctly specified.

**Assessment: Excellent. Ready to implement.**

### 2.6 Sprint Sequencing

The 14-sprint sequence is logically ordered. Each sprint's dependencies are correctly placed upstream:

- Authentication before everything
- Studies before Subjects
- Subjects before Visits
- Visits before Charts
- Task Engine before Business Rules
- Business Rules before Clinical Intelligence

This is the correct build order.

**Assessment: Correct.**

## 3. Architectural Risks

### RISK-01: Business Rule Engine — CRITICAL

**Severity: CRITICAL**

The Business Rule Engine is referenced in every module. It is the automation heart of the platform. Yet the execution schema is completely undefined. The `conditions` and `actions` fields are typed as JSONB with no documented structure.

**Impact:** Without a defined schema, the engine cannot be built correctly in Sprint 9. More critically, other sprints (5, 6, 4) that depend on rules firing correctly cannot be tested properly.

**Recommendation:**
Define the JSONB schema now, before Sprint 1:

```typescript
// Condition schema
interface RuleCondition {
  field: string; // e.g., "chart.days_pending", "document.expiration_date"
  operator: '>' | '<' | '>=' | '<=' | '==' | '!=' | 'in' | 'not_in' | 'within_days';
  value: number | string | string[] | null;
}

interface RuleConditionGroup {
  logic: 'AND' | 'OR';
  conditions: (RuleCondition | RuleConditionGroup)[];
}

// Action schema
interface RuleAction {
  type:
    | 'create_task'
    | 'update_status'
    | 'send_notification'
    | 'calculate_kpi'
    | 'write_audit'
    | 'call_ai_agent';
  payload: {
    title?: string;
    description?: string;
    priority?: 'critical' | 'high' | 'medium' | 'low';
    assigned_role?: string;
    due_date_offset_days?: number;
    status_value?: string;
    notification_type?: string;
    kpi_name?: string;
    agent_key?: string;
  };
}

// Full rule structure
interface BusinessRuleSchema {
  trigger: string; // e.g., "chart.created"
  conditions: RuleConditionGroup;
  actions: RuleAction[];
}
```

This schema must be frozen before Sprint 9 and documented as seed data for default rules.

### RISK-02: Visit Rescheduling Cascade

**Severity: HIGH**

When a user reschedules a visit with option `this_and_future_visits`, future visit dates must be recalculated. The spec does not define the algorithm.

**Questions:**

- Are future dates calculated as offset from the rescheduled visit's new date?
- Or are they recalculated from `subjects.baseline_date`?
- What if a future visit is `confirmed` (patient has appointment)?

**Recommendation:**
Always recalculate from `subjects.baseline_date`. If a future visit is `confirmed`, flag it as requiring manual review rather than auto-rescheduling. Never auto-reschedule `in_progress` or `completed` visits.

---

### RISK-03: Analytics KPI Calculation Timing

**Severity: HIGH**

The spec states KPIs are "recalculated incrementally" but defines no mechanism. A naive implementation (full recalculation on every page view) will fail at scale. A trigger-based approach (recalculate after every write) creates write amplification.

**Recommendation — Three-Tier Strategy:**

```
Tier 1 — Immediate (< 100ms): Simple counts updated in-place
  Example: pending_charts_count incremented when chart is created
  Mechanism: Service layer calls AnalyticsService.increment('pending_charts', site_id)

Tier 2 — Near-real-time (< 5 min): Aggregates recomputed per event
  Example: average_entry_time recalculated when chart is entered
  Mechanism: Service layer calls AnalyticsService.recalculate('avg_entry_time', context)

Tier 3 — Periodic (hourly): Complex multi-table aggregations
  Example: site_productivity_score, regulatory_compliance_score
  Mechanism: Supabase Edge Function on cron, writes to analytics_kpis
```

---

### RISK-04: N+1 Query Risk in Subject Profile

**Severity: HIGH**

`GET /api/subjects/:id` returns overview, visits, charts, timeline, notes, documents, history. A naive implementation with one query per section creates 7+ database queries per page load.

**Recommendation:**

- Implement lazy tab loading: only load tab data when the tab is activated.
- For "Overview" tab: one optimized JOIN query combining subject + recent visits + recent charts + open tasks.
- Use React Suspense boundaries per tab to show progressive loading.
- Never load all tabs on initial page render.

---

### RISK-05: Supabase Edge Functions Cold Start

**Severity: MEDIUM**

Protocol extraction (the most critical AI operation) calls a Supabase Edge Function. Edge Functions on Supabase have cold start latency (500ms–2s). For a PDF extraction that may take 15–30 seconds anyway, this is acceptable. However, the UX must manage this expectation.

**Recommendation:**

- Show a processing status on protocol upload: "Analyzing protocol... (this may take up to 60 seconds)"
- Implement async polling: upload triggers extraction, return a `request_id`, client polls `/api/ai/protocol/status/{request_id}` every 5 seconds.
- Store extraction status in `ai_requests.status`: `pending → processing → completed → failed`.

---

### RISK-06: Concurrent Write Conflicts

**Severity: MEDIUM**

Two scenarios create race conditions:

1. Two CRCs completing the same visit simultaneously → two charts created
2. Two Data Entry staff entering the same chart → conflicting updates

**Recommendation:**

- Add `UNIQUE(visit_id)` on `charts.visit_id` (GAP-REQ-01). The second chart creation fails at DB level with a unique violation, handled gracefully as `DUPLICATE_RECORD`.
- Add optimistic locking on charts: include `updated_at` in update payload; reject if `updated_at` doesn't match current DB value.

---

### RISK-07: OCR Accuracy on Clinical PDFs

**Severity: MEDIUM**

Clinical protocol PDFs are often:

- Scanned (not digital-native)
- Low resolution
- Multi-column layouts
- Tables with complex structures

Claude's vision capability handles these well for digital PDFs. For scanned PDFs, accuracy may drop.

**Recommendation:**

- Always present AI extraction with confidence scores.
- Every extracted field is editable before approval.
- Never auto-approve AI extractions (already specified, just reinforcing).
- For MVP, Claude vision is sufficient. Evaluate dedicated OCR services (AWS Textract) for v1.5 based on user feedback.

---

### RISK-08: Regulatory Health Score Gaming

**Severity: LOW**

The Regulatory Health Score (`100% = all required documents current`) depends on what's in `study_document_requirements`. If this table is under-populated, the score is artificially inflated.

**Recommendation:**

- Auto-populate `study_document_requirements` from `document_types.required_by_default = true` when study is activated.
- Regulatory Agent should add additional requirements detected in the protocol.
- Admin UI should show "Required documents not yet configured" warning if requirements table is empty for an active study.

## 4. Database Design Review

### 4.1 Schema Evaluation — Table by Table

**companies** ✓ Clean. Add `country` and `primary_contact_email` fields for enterprise support.

**profiles** ✓ Correct mapping to auth.users. The `avatar_file_id` FK is correct (references files table).

**roles / permissions / role_permissions** ✓ Standard RBAC. Correctly designed.

**sites** ✓ Clean. Consider adding `country` for international multi-site studies.

**studies** ✓ `ai_generated boolean` is good metadata. Consider adding `target_enrollment integer` for analytics.

**visit_templates / visit_template_items** ✓ The versioning model (one template per study, new version = new record) is correct. Never overwrite approved templates.

**subjects** ⚠️ Missing `UNIQUE(study_id, subject_number)` constraint. See GAP-REQ-02.

**visits** ✓ `visit_template_item_id` nullable for unscheduled — correct. `window_start/window_end` as computed fields — correct.

**charts** ⚠️ Missing `UNIQUE(visit_id)` — prevents duplicate charts. See GAP-REQ-01. Add `priority_score numeric` for queue sorting.

**tasks** ✓ `source_record_type + source_record_id` polymorphic reference is correct (same pattern as `file_links`).

**files** ⚠️ Missing site isolation — handled via `file_links`. Must document this clearly in RLS policies.

**business_rules** ⚠️ `conditions jsonb` and `actions jsonb` lack defined schemas. Critical gap.

**ai_requests / ai_responses** ✓ Correct pattern. Add `rejected_at timestamptz` and `rejected_by uuid` to `ai_responses`.

**audit_logs** ✓ Comprehensive. Add `session_id text` for correlating related actions within one user session.

### 4.2 Missing Tables Summary

```
notifications                → in-app notification storage
notification_preferences     → per-user, per-event-type settings
user_invitations             → invitation token tracking
```

See GAP_ANALYSIS.md sections GAP-DB-01 and GAP-DB-02.

### 4.3 Missing Indexes Summary

The spec defines basic indexes in DATABASE_Part_01 and _02, but critical operational indexes are missing. See GAP_ANALYSIS.md for the complete list. The most critical:

```sql
-- These are required for RLS policy performance
idx_subjects_site, idx_visits_site, idx_charts_site, idx_tasks_site

-- Required for chart queue ordering
idx_charts_priority_score (once priority_score column added)

-- Required for regulatory expiration queries
idx_regulatory_docs_expiration
```

## 5. API Design Review

### 5.1 Positive Observations

- Standard response envelope (`{ success, data, message }`) is consistent and correct.
- Error codes are typed and comprehensive.
- API route ordering in the spec matches the sprint dependency chain.
- Separation of `GET /api/charts` (all charts) from `GET /api/charts/queue` (prioritized queue) is a clean design decision.
- Signed URL endpoint (`GET /api/files/:id/signed-url`) correctly separates metadata from access.

### 5.2 API Gaps

**Missing: `GET /api/subjects/:id/visits`**
The API spec does not define an endpoint for loading a subject's visits. This is needed for the Subject Profile "Visits" tab. Either include visits in the `GET /api/subjects/:id` response (via query param `?include=visits`) or add a dedicated endpoint.

**Missing: `GET /api/analytics/kpis?site_id=...`**
KPI endpoint should support site-level filtering to show per-site dashboards.

**Missing: Bulk operations**
No bulk task completion, no bulk subject status change. These are v1.5 features per ROADMAP but should be architected with extensibility in mind (batch operation pattern).

**Missing: `PATCH /api/regulatory/documents/:id`**
There is an archive and version endpoint but no general update endpoint for correcting metadata before AI review is approved.

### 5.3 REST vs. Server Actions

The spec defines REST API routes, but Next.js App Router prefers Server Actions for mutations. Recommendation:

```
Use API Routes for:
  - External integrations (future)
  - Edge Functions to call back
  - Complex file operations
  - Operations requiring custom HTTP status codes

Use Server Actions for:
  - Form submissions
  - Simple CRUD mutations
  - Status changes from UI buttons
```

This reduces client-server roundtrips and simplifies the architecture for pure-UI interactions.

## 6. Security Model Review

### 6.1 Positive Observations

The security model is correctly designed. The layered enforcement (DB → API → Service → Storage) is the correct approach. The prohibition of frontend-only security enforcement is explicitly stated and correct.

### 6.2 Security Concerns

**Concern 1: RLS Helper Function Performance**
`current_company_id()` and `has_permission()` are called on every RLS policy evaluation. If these functions execute subqueries on every row, they create significant performance overhead under load.

**Mitigation:**

- Use `SECURITY DEFINER STABLE` on all helper functions (already specified).
- Consider caching company_id in the JWT claims via Supabase custom claims, so `current_company_id()` is a simple JWT claim read rather than a table lookup.
- Index `profiles(id)` with `company_id` included for the lookup.

**Concern 2: Service Role Key in Edge Functions**
Edge Functions use the service role key. If an Edge Function has a vulnerability (injection, logic flaw), the service role key provides full database access without RLS.

**Mitigation:**

- Keep Edge Functions minimal and purpose-specific.
- Validate all inputs in Edge Functions before any database operations.
- Log all service role operations in `audit_logs` with a system user identifier.

**Concern 3: Signed URL Replay**
A signed URL with a 1-hour expiry can be captured and replayed within that window.

**Mitigation:**

- Use 1-hour expiry (already recommended).
- For sensitive documents, add IP binding if Supabase supports it.
- For download links in emails, use 15-minute expiry.
- Log every signed URL generation in `audit_logs`.

**Concern 4: Mass Document Exfiltration**
A compromised account with `view_all_sites` permission could exfiltrate all company documents.

**Mitigation:**

- Rate limit `GET /api/files/:id/signed-url` per user per hour.
- Alert on excessive signed URL generation (>50 in 10 minutes → flag for review).
- Track in `audit_logs`: user, file_id, signed URL generation time.

## 7. AI Architecture Review

### 7.1 Positive Observations

The AI architecture is well-conceived. The "never write, always propose" model is correct for clinical software. The structured JSON output requirement prevents ambiguous AI responses. The confidence score requirement enables UI-level uncertainty disclosure.

### 7.2 AI Architecture Concerns

**Concern 1: Context Window Limits**
Large protocol PDFs may exceed Claude's context window. A 300-page protocol document cannot be fully processed in a single API call.

**Mitigation:**

- Chunk large documents: split into sections (schedule of assessments, ICF, main protocol text).
- Process each section independently.
- Merge results in the Protocol Agent service.
- Track which sections were processed vs. skipped in `study_ai_extractions.extracted_data`.

**Concern 2: AI Cost at Scale**
Running the Protocol Agent on every protocol upload, Regulatory Agent on every document, and Subject Agent reviews generates significant Claude API costs.

**Mitigation:**

- Cache AI results: if the same document (by checksum) is uploaded again, return cached extraction.
- Implement AI request budgets per company per month (configurable in Settings).
- For the MVP, cost is not a constraint — document it for v1.5 planning.

**Concern 3: Hallucination in Visit Templates**
If the Protocol Agent invents visit timing not present in the protocol, clinical consequences could be significant.

**Mitigation:**

- Every uncertain extraction is marked `"confidence": "low"` with a visual warning in the UI.
- The AIReviewPanel shows the source text excerpt alongside each extracted value.
- No visit template can be approved without explicit human review of every item.
- Users must check a box confirming they've reviewed AI-suggested fields before approval.

**Concern 4: No Training Agent UI**
The specification references a Training Agent in PROMPTS.md but provides no UI spec and no CI_XX document for it. Including it in the `ai_agents` seed data without a UI creates a dead record.

**Recommendation:** Exclude Training Agent from MVP. Add to ROADMAP v1.5.

## 8. Scalability Review

### 8.1 Projected Data Volumes (Year 1)

Estimated at launch for a mid-size CRO with 5 companies, 20 sites, 50 studies:

```
subjects:          ~5,000 records
visits:            ~50,000 records
charts:            ~40,000 records
regulatory_docs:   ~10,000 records
files:             ~15,000 records
audit_logs:        ~500,000 records (growing ~50k/month)
tasks:             ~100,000 records (historical)
analytics_kpis:    ~2,000 records (aggregated)
```

These volumes are manageable in Supabase PostgreSQL with proper indexing. No architectural changes are needed for Year 1.

### 8.2 Scaling Concerns for Year 2+ (GAP-SCALE analysis)

**audit_logs table:** Will grow to millions of rows. See GAP-SCALE-01 for archival strategy.

**Full-text search:** At 100k+ documents, `tsvector` on the `files` table with GIN indexes is sufficient. At 1M+ documents, evaluate dedicated search (Typesense, Algolia, pg_vectorize) for v2.0.

**analytics_kpis:** The current design works for MVP. For v2.0 with complex cross-study analytics, consider a dedicated OLAP solution (ClickHouse, BigQuery) for the Analytics module.

**Multi-company growth:** The architecture correctly supports unlimited companies from day one. No per-company partitioning is needed at current scale.

## 9. UI/UX Architecture Review

### 9.1 Positive Observations

The three-click rule is correctly specified. The Command Palette (`Ctrl+K`) is a productivity feature that reduces navigation depth. The Task Center as the operational home screen is the right design for CRC and Data Entry users.

The role-based dashboard customization (CRC sees visits/tasks, CEO sees KPIs/analytics) is correctly designed and aligned with how clinical sites actually operate.

### 9.2 UI Architecture Concerns

**Concern 1: Dashboard Performance**
The dashboard loads KPI cards, alerts, AI suggestions, visit calendar, and charts simultaneously. On slow connections, this creates perceived slowness.

**Recommendation:**

- Render the page shell immediately (Sidebar, TopBar).
- Use React Suspense with skeleton loaders for each dashboard section.
- Load AI suggestions last (lowest priority).
- Target: page is interactive within 1 second; all widgets loaded within 3 seconds.

**Concern 2: Mobile Experience**
The spec says mobile shows "Executive dashboards and basic actions only." The CRC role needs access on tablet for bedside visit completion. Ensure the Visit Completion flow works on tablet (landscape mode, minimum 768px width).

**Concern 3: Visit Calendar Complexity**
The calendar displays patient visits (from `visits` table) and manual events (from `calendar_events`). Both must be filtered by site. The query combining both sources requires a UNION or two separate queries with client-side merge.

**Recommendation:** Create a `GET /api/calendar/events` endpoint that returns both types (already specified in the API). The service layer handles the union query. The frontend sees a single normalized event stream.

## 11. Final Verdict

ClinicalOS is architecturally sound and ready to build. The specification is detailed enough to begin implementation without significant ambiguity in most areas.

The two critical items that must be resolved before a single line of production code is written:

**Critical #1:** Business Rule Engine JSONB schema (GAP-BL-01)
**Critical #2:** Missing database tables — notifications and user_invitations (GAP-DB-01, GAP-DB-02)

Everything else can be addressed incrementally during the appropriate sprint.

The architecture is enterprise-grade, security-correct, and correctly scoped for MVP delivery. Proceed to implementation.
