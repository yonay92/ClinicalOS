# ClinicalOS — Coding Standards & Conventions

Version: 1.0
Generated by: Lead Software Architect
Status: Mandatory for all sprints

---

## 1. Guiding Principles

Every line of code in ClinicalOS must satisfy these principles:

1. **Explicit over implicit** — Name things clearly; avoid abbreviations.
2. **Predictable over clever** — Simple code that reads like the spec is better than clever code.
3. **Typed, always** — TypeScript strict mode, no `any`.
4. **Secure by default** — Every write validates permissions; every file access uses signed URLs.
5. **Testable by design** — Services are pure functions that take context; they don't reach for globals.

## 2. TypeScript Standards

### 2.1 Strict Mode

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

### 2.2 No `any`

```typescript
// ❌ Forbidden
const data: any = await fetchData();
function process(input: any) {}

// ✓ Correct
const data: Subject = await fetchData();
function process(input: CreateSubjectInput): Promise<Subject> {}
```

### 2.3 Explicit Return Types on All Public Functions

```typescript
// ❌ Implicit return type
async function getSubject(id: string) {
  return supabase.from('subjects').select('*').eq('id', id);
}

// ✓ Explicit return type
async function getSubject(id: string): Promise<Subject | null> {
  const { data } = await supabase.from('subjects').select('*').eq('id', id).single();
  return data;
}
```

### 2.4 Type Definitions Pattern

All domain types live in `/types`. Each domain file exports:

- The database row type (matches the DB schema exactly)
- Input types for create/update operations
- Response types for API payloads

```typescript
// types/subjects.ts

// Database row — mirrors the DB schema
export type Subject = {
  id: string;
  company_id: string;
  site_id: string;
  study_id: string;
  subject_number: string;
  initials: string | null;
  status: SubjectStatus;
  screening_date: string | null;
  baseline_date: string | null;
  randomization_date: string | null;
  end_of_study_date: string | null;
  created_by: string;
  created_at: string;
  updated_at: string;
};

// Status enum
export type SubjectStatus =
  | 'pre_screening'
  | 'screening'
  | 'screen_failed'
  | 'randomized'
  | 'active'
  | 'completed'
  | 'early_terminated'
  | 'lost_to_follow_up';

// Input for creating
export type CreateSubjectInput = {
  site_id: string;
  study_id: string;
  subject_number: string;
  initials?: string;
  screening_date?: string;
};

// Input for status change
export type ChangeSubjectStatusInput = {
  subject_id: string;
  new_status: SubjectStatus;
  reason?: string;
};
```

### 2.5 Zod Schemas for Runtime Validation

Every API input has a matching Zod schema in `lib/utils/validation.ts`:

```typescript
// lib/utils/validation.ts
import { z } from 'zod';

export const createSubjectSchema = z.object({
  site_id: z.string().uuid(),
  study_id: z.string().uuid(),
  subject_number: z.string().min(1).max(50).trim(),
  initials: z.string().max(10).trim().optional(),
  screening_date: z.string().date().optional(),
});

export type CreateSubjectSchema = z.infer<typeof createSubjectSchema>;
```

The TypeScript input type and the Zod schema must always be kept in sync. Prefer deriving the TypeScript type from the Zod schema (`z.infer`) to prevent drift.

## 4. Service Layer Standards

### 4.1 Service Structure

Every service is a class or module in `/services/{domain}/{DomainService}.ts`:

```typescript
// services/subjects/SubjectService.ts
import { createServerSupabaseClient } from '@/lib/supabase/server';
import { BusinessRuleEngine } from '@/services/business-rules/BusinessRuleEngine';
import { AuditService } from '@/services/audit/AuditService';
import { PermissionService } from '@/services/permissions/PermissionService';

export const SubjectService = {
  async createSubject(
    input: CreateSubjectInput,
    ctx: { user: Profile; company: Company },
  ): Promise<Subject> {
    const supabase = createServerSupabaseClient();

    // 1. Validate permissions
    await PermissionService.requirePermission(ctx.user, 'create_subject');

    // 2. Validate business rules (pre-condition checks)
    await validateStudyActive(input.study_id, ctx.company.id);
    await validateStudySiteExists(input.study_id, input.site_id);
    await validateApprovedTemplateExists(input.study_id);

    // 3. Execute the write
    const { data: subject, error } = await supabase
      .from('subjects')
      .insert({ ...input, company_id: ctx.company.id, created_by: ctx.user.id })
      .select()
      .single();

    if (error) throw new DatabaseError(error.message);

    // 4. Fire business rules (post-write automations)
    await BusinessRuleEngine.evaluate('subject.created', {
      subject_id: subject.id,
      study_id: subject.study_id,
      site_id: subject.site_id,
      company_id: subject.company_id,
    });

    // 5. Write audit log
    await AuditService.log({
      user_id: ctx.user.id,
      company_id: ctx.company.id,
      site_id: subject.site_id,
      action: 'create_subject',
      module: 'subjects',
      record_type: 'subject',
      record_id: subject.id,
      new_value: subject,
    });

    return subject;
  },
};
```

### 4.2 Context Pattern

Services never reach for the current user from globals. They receive `ctx` as a parameter:

```typescript
// ✓ Correct — context is explicit
async createSubject(input: CreateSubjectInput, ctx: { user: Profile; company: Company })

// ❌ Forbidden — implicit global access
async createSubject(input: CreateSubjectInput) {
  const user = getCurrentUser(); // Never do this in a service
}
```

This makes services testable: in tests, pass a mock context.

### 4.3 Company ID Is Never Trusted from Client

```typescript
// ❌ CRITICAL SECURITY VIOLATION — never do this
const company_id = request.body.company_id;

// ✓ Always derive company_id from the authenticated session
const company_id = ctx.company.id; // comes from resolveAuthContext(request)
```

## 5. API Route Standards

### 5.1 Route Handler Template

```typescript
// app/api/subjects/route.ts
import { NextRequest } from 'next/server';
import { resolveAuthContext } from '@/lib/api/middleware';
import { successResponse, errorResponse } from '@/lib/api/response';
import { createSubjectSchema } from '@/lib/utils/validation';
import { SubjectService } from '@/services/subjects/SubjectService';

export async function POST(request: NextRequest) {
  // 1. Resolve auth context
  const authResult = await resolveAuthContext(request);
  if (!authResult.ok) return errorResponse('UNAUTHORIZED', 401);
  const { user, company } = authResult;

  // 2. Parse and validate body
  const body = await request.json();
  const validated = createSubjectSchema.safeParse(body);
  if (!validated.success) {
    return errorResponse('VALIDATION_ERROR', 400, { issues: validated.error.issues });
  }

  // 3. Call service
  try {
    const subject = await SubjectService.createSubject(validated.data, { user, company });
    return successResponse(subject, 'Subject created successfully', 201);
  } catch (error) {
    if (error instanceof PermissionDeniedError) return errorResponse('FORBIDDEN', 403);
    if (error instanceof BusinessRuleError)
      return errorResponse('BUSINESS_RULE_FAILED', 422, { message: error.message });
    if (error instanceof DuplicateRecordError) return errorResponse('DUPLICATE_RECORD', 409);
    throw error; // Let Next.js error handler catch unexpected errors
  }
}
```

### 5.2 Never in Route Handlers

```
❌ No database queries
❌ No business logic
❌ No permission checking
❌ No audit logging
❌ No direct Supabase client calls
```

All of the above belong in the service layer.

## 7. Database and RLS Standards

### 7.1 Supabase Client Usage

```typescript
// lib/supabase/server.ts — for server components and API routes
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export function createServerSupabaseClient() {
  const cookieStore = cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { get: (name) => cookieStore.get(name)?.value } },
  );
}

// lib/supabase/client.ts — for client components (realtime, hooks)
import { createBrowserClient } from '@supabase/ssr';

export function createBrowserSupabaseClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
  );
}

// lib/supabase/admin.ts — ONLY for Edge Functions and service role operations
// ⚠️ NEVER import this in any file under /app or /components
import { createClient } from '@supabase/supabase-js';

export function createAdminSupabaseClient() {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!,
  );
}
```

### 7.2 Query Patterns

```typescript
// ✓ Always select specific columns, never '*' in production code
const { data } = await supabase
  .from('subjects')
  .select('id, subject_number, status, study_id, site_id')
  .eq('company_id', company.id) // Redundant with RLS but adds clarity
  .eq('study_id', studyId)
  .order('subject_number');

// ✓ Nested selects for related data
const { data } = await supabase
  .from('subjects')
  .select(
    `
    id, subject_number, status,
    visits(id, visit_name, status, target_date),
    subject_notes(id, note, created_at)
  `,
  )
  .eq('id', subjectId)
  .single();

// ❌ Never select *
const { data } = await supabase.from('subjects').select('*'); // Forbidden
```

```sql
-- Migration: 004_subjects_visits_calendar.sql
-- Description: Creates subjects, visits, and calendar events tables
-- Depends on: 003_studies_visit_templates.sql

-- ============================================================
-- SUBJECTS
-- ============================================================

CREATE TABLE subjects (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),

  site_id uuid NOT NULL REFERENCES sites(id) ON DELETE RESTRICT,
  study_id uuid NOT NULL REFERENCES studies(id) ON DELETE RESTRICT,
  subject_number text NOT NULL,

  screening_date date,
  baseline_date date,
  randomization_date date,
  end_of_study_date date,
  created_by uuid NOT NULL REFERENCES profiles(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),

CREATE INDEX idx_subjects_site ON subjects(site_id);
CREATE INDEX idx_subjects_study ON subjects(study_id);
CREATE INDEX idx_subjects_status ON subjects(status);

CREATE POLICY subjects_company_isolation ON subjects
  FOR ALL USING (company_id = current_company_id());

CREATE POLICY subjects_site_isolation ON subjects
  FOR SELECT USING (
    company_id = current_company_id()
    AND (
      site_id IN (SELECT site_id FROM user_sites WHERE user_id = auth.uid())
      OR has_permission('view_all_sites')
    )
  );

-- Updated_at trigger
CREATE TRIGGER subjects_updated_at
  BEFORE UPDATE ON subjects
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

-- ============================================================
-- ROLLBACK (comment — not executed automatically)
-- DROP TABLE IF EXISTS subjects CASCADE;
-- ============================================================
```

## 8. Error Handling Standards

### 8.1 Domain Error Classes

```typescript
// lib/api/errors.ts

export class AppError extends Error {
  constructor(
    message: string,
    public readonly code: string,
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export class PermissionDeniedError extends AppError {
  constructor(permission: string) {
    super(`Permission denied: ${permission}`, 'FORBIDDEN');
  }
}

export class BusinessRuleError extends AppError {
  constructor(message: string) {
    super(message, 'BUSINESS_RULE_FAILED');
  }
}

export class DuplicateRecordError extends AppError {
  constructor(field: string) {
    super(`Duplicate record: ${field} already exists`, 'DUPLICATE_RECORD');
  }
}

export class NotFoundError extends AppError {
  constructor(resource: string) {
    super(`${resource} not found`, 'NOT_FOUND');
  }
}

export class DatabaseError extends AppError {
  constructor(message: string) {
    super(`Database operation failed: ${message}`, 'INTERNAL_ERROR');
  }
}

export class AIReviewRequiredError extends AppError {
  constructor(request_id: string) {
    super(`AI response requires review before applying`, 'AI_REVIEW_REQUIRED');
    this.request_id = request_id;
  }
  request_id: string;
}
```

### 8.2 Never Expose Internals

```typescript
// ❌ Leaks internal details
return Response.json({ error: 'relation "subjects" does not exist' }, { status: 500 });

// ✓ User-friendly message, internal details logged
logger.error('Database error in SubjectService.createSubject', { error, context });
return errorResponse('INTERNAL_ERROR', 500);
```

### 8.3 Logging Requirements

Every unexpected error must be logged with:

```typescript
logger.error('Operation failed', {
  error: error.message,
  stack: error.stack,
  user_id: ctx?.user?.id,
  company_id: ctx?.company?.id,
  operation: 'SubjectService.createSubject',
  input: sanitizeForLog(input), // Remove any PII
});
```

## 9. Audit Log Standards

### 9.1 What Must Be Audited

Every service write that modifies operational data must call `AuditService.log()`. No exceptions for:

```
User created / deactivated / role changed
Study created / activated / closed
Protocol uploaded / AI approved
Subject created / status changed
Visit completed / cancelled / rescheduled
Chart marked entered in EDC
Document uploaded / archived / versioned
Business rule created / modified / executed (failure)
AI suggestion approved / rejected
Permission changes
Login / logout
```

### 9.2 Audit Log Pattern

```typescript
await AuditService.log({
  user_id: ctx.user.id,
  company_id: ctx.company.id,
  site_id: record.site_id, // null if not site-specific
  action: 'mark_chart_entered',
  module: 'charts',
  record_type: 'chart',
  record_id: chart.id,
  old_value: { status: 'in_progress', entered_in_edc_date: null },
  new_value: { status: 'entered_in_edc', entered_in_edc_date: new Date().toISOString() },
});
```

### 9.3 Audit Logs Are Immutable

The `audit_logs` table must have NO UPDATE and NO DELETE RLS policies. Ever:

```sql
CREATE POLICY audit_logs_insert_only ON audit_logs
  FOR INSERT WITH CHECK (company_id = current_company_id());

-- No UPDATE policy
-- No DELETE policy
-- SELECT: restricted by permission
CREATE POLICY audit_logs_read ON audit_logs
  FOR SELECT USING (
    company_id = current_company_id()
    AND has_permission('view_audit_logs')
  );
```

## 10. Testing Standards

### 10.1 Unit Test Pattern (Vitest)

```typescript
// tests/unit/services/SubjectService.test.ts
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { SubjectService } from '@/services/subjects/SubjectService';

describe('SubjectService.createSubject', () => {
  const mockCtx = {
    user: { id: 'user-1', company_id: 'company-1', permissions: ['create_subject'] },
    company: { id: 'company-1' },
  };

  it('creates subject and generates visits when baseline_date provided', async () => {
    // Arrange
    const input = { site_id: 'site-1', study_id: 'study-1', subject_number: '001-001' };

    // Act
    const result = await SubjectService.createSubject(input, mockCtx);

    // Assert
    expect(result.subject_number).toBe('001-001');
    expect(result.status).toBe('screening');
  });

  it('throws BusinessRuleError when study is closed', async () => {
    // Arrange — mock closed study
    vi.spyOn(StudyRepository, 'findById').mockResolvedValue({ status: 'closed' });

    // Act + Assert
    await expect(SubjectService.createSubject(input, mockCtx)).rejects.toThrow(BusinessRuleError);
  });

  it('throws PermissionDeniedError when user lacks create_subject permission', async () => {
    const noPermCtx = { ...mockCtx, user: { ...mockCtx.user, permissions: [] } };
    await expect(SubjectService.createSubject(input, noPermCtx)).rejects.toThrow(
      PermissionDeniedError,
    );
  });
});
```

## 14. Security Anti-Patterns to Avoid

The following patterns are forbidden and will be rejected in code review:

```typescript
// ❌ Trusting client-provided company_id
const { company_id } = req.body;
await supabase.from('subjects').insert({ company_id, ... }); // CRITICAL VULNERABILITY

// ❌ Using service role client in a component or API route
import { createAdminSupabaseClient } from '@/lib/supabase/admin';  // Forbidden outside Edge Functions

// ❌ Disabling RLS for convenience
await supabase.rpc('disable_rls'); // Never acceptable

// ❌ Generating permanent public URLs for files
const url = `${supabaseUrl}/storage/v1/object/public/regulatory/${fileName}`; // Forbidden

// ❌ Logging sensitive data
logger.info('User logged in', { password: input.password }); // Never log passwords or PII

// ❌ Skipping permission check
async function markChartEntered(chartId: string) {
  // Missing: await PermissionService.requirePermission(user, 'mark_chart_entered');
  await supabase.from('charts').update({ status: 'entered_in_edc' }).eq('id', chartId);
}
```

## 15. Definition of Done — Code Quality

A feature is not done until:

```
□ TypeScript: zero type errors (tsc --noEmit passes)
□ Lint: zero ESLint warnings or errors
□ Format: Prettier formatted (pnpm format run)
□ Tests: unit tests pass for all new service methods
□ Tests: RLS tests pass for all new tables
□ Tests: E2E test covers the primary user flow
□ Security: permission check in every service write method
□ Security: audit log written for every critical action
□ Security: company_id never taken from client input
□ API: Zod validation on all route inputs
□ No 'any' types
□ No console.log statements (use logger)
□ No TODO or placeholder comments
□ No hardcoded UUIDs or company-specific values
```
