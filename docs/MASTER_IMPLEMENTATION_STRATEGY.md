## 1. Strategic Overview

ClinicalOS is a 14-sprint enterprise SaaS platform. This document defines the execution philosophy, risk management approach, integration architecture, and cross-cutting concerns that govern how every sprint is built.

This document is the strategic layer above IMPLEMENTATION_PLAN.md. Where the Implementation Plan answers "what to build and in what order," this document answers "how to build it correctly across all sprints."

---

## 2. Core Build Philosophy

### 2.1 Security-First, Not Security-Later

Security is enforced at four independent layers. Every sprint must implement all four:

```
Layer 1: Database    → RLS policies on every table
Layer 2: API         → Permission + company + site validation on every route
Layer 3: Service     → Business rule validation before every write
Layer 4: Storage     → Private buckets, signed URLs, no permanent public access
```

The frontend is UX. It is never the security boundary.

**Sprint rule:** No sprint ships without RLS policies on its tables. RLS is written in the same PR as the migration.

---

### 2.2 Layered Architecture — No Shortcuts

Every feature must flow through the exact same layer stack:

```
UI Component
  → Server Action / API Route
    → Service
      → Business Rule Engine (if write operation)
        → Supabase Repository (RLS-enforced)
          → Audit Service (if critical action)
            → Analytics Service (if KPI-affecting)
              → Notification Service (if user-facing event)
```

**Non-negotiable:** React components never call Supabase directly for writes. The service layer is always the intermediary. This is enforced by code review and ESLint rules.

---

### 2.3 Automation-First Data Flows

Every state transition in ClinicalOS is an event that the Business Rule Engine evaluates. No transition is silent.

```
Every Write Operation triggers:
1. Service validates permissions
2. Service executes the write
3. BusinessRuleEngine.evaluate(trigger, context)
4. AuditService.log(action, before, after)
5. AnalyticsService.recalculate(affected_kpis)
6. NotificationService.notify(affected_users)
```

This pipeline is implemented inside each service method. It is not optional.

---

### 2.4 AI Integration Model

AI is a proposer, never an executor. The integration model is fixed across all agents:

```
1. User or Business Rule triggers AI request
2. AI request stored in ai_requests (status = 'pending')
3. Edge Function calls AI provider (Claude claude-sonnet-4-6)
4. AI response stored in ai_responses (requires_review = true)
5. User reviews the proposed changes in AIReviewPanel
6. User approves → Business Rules execute the actual write
7. User rejects → ai_responses.rejected_at set, no data changed
8. audit_logs records: who approved, when, what was applied
```

This model applies to ALL agents: Protocol, Regulatory, Subject, Data, Analytics, Executive.

## 3. Database Strategy

### 3.1 Migration Discipline

```
Every schema change = one migration file.
Migration files are numbered sequentially.
Migrations are never edited after being run in any environment.
Every migration is reversible (include a rollback comment).
Migrations are tested in local Supabase before staging.
```

Migration file naming:

```
001_companies_sites_users.sql
002_roles_permissions.sql
003_studies_visit_templates.sql
004_subjects_visits_calendar.sql
005_charts_tasks_analytics.sql
006_regulatory_files.sql
007_business_rules_ai_audit.sql
008_notifications.sql
009_indexes_functions.sql
```

### 3.2 RLS Policy Strategy

Every table gets RLS enabled the moment it is created. The policy pattern is consistent:

```sql
-- Company isolation (all tables)
CREATE POLICY "company_isolation" ON {table}
  FOR ALL USING (company_id = current_company_id());

-- Site isolation (operational tables)
CREATE POLICY "site_isolation" ON {table}
  FOR ALL USING (
    company_id = current_company_id()
    AND (
      site_id IN (SELECT site_id FROM user_sites WHERE user_id = auth.uid())
      OR has_permission('view_all_sites')
    )
  );

-- Permission-gated writes (sensitive tables)
CREATE POLICY "permission_write" ON {table}
  FOR INSERT WITH CHECK (
    company_id = current_company_id()
    AND has_permission('{required_permission}')
  );
```

### 3.3 Helper Functions (Sprint 1 — Required by All Sprints)

These four SQL functions are the foundation of all RLS policies. They MUST be created before any other table's RLS policy is written:

```sql
-- Returns the company_id of the authenticated user
CREATE OR REPLACE FUNCTION current_company_id() RETURNS uuid AS $$
  SELECT company_id FROM profiles WHERE id = auth.uid();
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Returns the site_ids accessible to the authenticated user
CREATE OR REPLACE FUNCTION current_user_sites() RETURNS SETOF uuid AS $$
  SELECT site_id FROM user_sites WHERE user_id = auth.uid();
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Checks if the authenticated user has a specific permission
CREATE OR REPLACE FUNCTION has_permission(permission_key text) RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1
    FROM user_roles ur
    JOIN role_permissions rp ON rp.role_id = ur.role_id
    JOIN permissions p ON p.id = rp.permission_id
    WHERE ur.user_id = auth.uid()
    AND p.key = permission_key
    AND rp.allowed = true
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Checks if the authenticated user can access a specific site
CREATE OR REPLACE FUNCTION can_access_site(target_site_id uuid) RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1 FROM user_sites
    WHERE user_id = auth.uid()
    AND site_id = target_site_id
  ) OR has_permission('view_all_sites');
$$ LANGUAGE sql SECURITY DEFINER STABLE;
```

### 3.4 Soft Delete Strategy

ClinicalOS does not hard-delete operational records. The strategy by table type:

```
Subject records:    status = 'early_terminated' or 'lost_to_follow_up'
Visit records:      status = 'cancelled'
Calendar events:    status = 'cancelled'
Document records:   status = 'archived'
Study records:      status = 'closed' or 'archived'
Task records:       status = 'cancelled'
User records:       status = 'inactive' or 'suspended'
Files:             Never deleted (preserve for audit)
AI responses:       Never deleted (preserve for audit)
Audit logs:         Never deleted, ever
```

Business rules may query `WHERE status NOT IN ('cancelled', 'archived', 'inactive')` to work on active records only.

## 4. API Strategy

### 4.1 Route Handler Pattern

Every API route follows the exact same pattern. No exceptions:

```typescript
// /app/api/{module}/{route}/route.ts

export async function POST(request: Request) {
  // 1. Authenticate
  const { user, company } = await resolveAuthContext(request);
  if (!user) return errorResponse('UNAUTHORIZED', 401);

  // 2. Validate payload
  const body = await request.json();
  const validated = schema.safeParse(body);
  if (!validated.success) return errorResponse('VALIDATION_ERROR', 400, validated.error);

  // 3. Call service (all business logic lives here)
  const result = await SomeService.doSomething(validated.data, { user, company });

  // 4. Return standard response
  return successResponse(result);
}
```

Route handlers contain zero business logic. They are HTTP adapters to the service layer.

### 4.2 Standard Response Format

All API responses use these two helpers from `lib/api/response.ts`:

```typescript
// Success
successResponse(data, message?, status = 200)
// Returns: { success: true, data, message }

// Error
errorResponse(code, status, details?)
// Returns: { success: false, error: { code, message, details } }
```

Error codes are the constants defined in `lib/api/errors.ts`:
`UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `VALIDATION_ERROR`,
`DUPLICATE_RECORD`, `BUSINESS_RULE_FAILED`, `AI_REVIEW_REQUIRED`,
`FILE_UPLOAD_FAILED`, `INTERNAL_ERROR`

### 4.3 Payload Validation

All API payloads are validated with Zod schemas defined in `lib/utils/validation.ts`:

```typescript
// Example: Create subject schema
export const createSubjectSchema = z.object({
  site_id: z.string().uuid(),
  study_id: z.string().uuid(),
  subject_number: z.string().min(1).max(50),
  initials: z.string().max(10).optional(),
  screening_date: z.string().date().optional(),
});
```

Schemas live in `lib/utils/validation.ts`, not inside route handlers. They are reused by both API routes and Server Actions.

## 5. Frontend Strategy

### 5.1 Server vs. Client Components

```
Server Components (default):
  - Page-level data fetching
  - Layouts
  - Static content
  - Tables and lists (initial load)

Client Components (use sparingly):
  - Forms with controlled inputs
  - Real-time subscriptions (Supabase realtime)
  - Command Palette (keyboard shortcuts)
  - Charts and interactive analytics
  - Drawers and Modals
```

Mark client components explicitly with `'use client'`. Default to server.

### 5.2 Data Fetching Pattern

```typescript
// Server component (RSC) — preferred for reads
async function StudyList() {
  const studies = await StudyService.getList({ user: await getServerUser() });
  return <DataTable data={studies} />;
}

// Client component — for interactive/realtime features
'use client';
function TaskCenter() {
  const { tasks, loading } = useTaskRealtime();
  return <TaskList tasks={tasks} loading={loading} />;
}
```

### 5.3 Server Actions for Mutations

Form submissions and mutations use Next.js Server Actions, not client-side fetch:

```typescript
// features/subjects/actions/subjectActions.ts
'use server';
export async function createSubjectAction(formData: FormData) {
  const validated = createSubjectSchema.safeParse(Object.fromEntries(formData));
  if (!validated.success) return { error: validated.error };
  return await SubjectService.createSubject(validated.data, { user: await getServerUser() });
}
```

### 5.4 Permission-Aware UI

Components check permissions before rendering actions:

```typescript
// hooks/usePermissions.ts
export function usePermissions() {
  const { user } = useAuth();
  return {
    can: (permission: string) => user?.permissions.includes(permission) ?? false,
  };
}

// Usage in component
const { can } = usePermissions();
{can('mark_chart_entered') && <Button onClick={markEntered}>Mark Entered in EDC</Button>}
```

Permissions are loaded once at login and stored in the session context. They are NOT refetched on every page (a session refresh handles updates).

## 6. Business Rule Engine Strategy

### 6.1 Engine Integration Points

The BusinessRuleEngine is called from inside service methods, not from API routes or components:

```typescript
// Inside SubjectService.createSubject()
await subjectRepository.insert(subjectData);           // 1. Write
await BusinessRuleEngine.evaluate('subject.created', { // 2. Evaluate rules
  subject_id: newSubject.id,
  study_id: subjectData.study_id,
  site_id: subjectData.site_id,
});
await AuditService.log('create_subject', ...);          // 3. Audit
```

### 6.2 Trigger Event Catalog

Every trigger has a string key. Services fire specific trigger keys:

```
subject.created
subject.status_changed
visit.completed
visit.missed
visit.out_of_window
chart.created
chart.entered_in_edc
chart.overdue
document.uploaded
document.expiring
document.expired
study.activated
study.closed
task.created
task.completed
protocol.uploaded
protocol.amendment
```

### 6.3 Rule Evaluation Algorithm

```typescript
// BusinessRuleEngine.ts
async evaluate(trigger: string, context: Record<string, unknown>) {
  const rules = await getActiveRulesByTrigger(trigger);
  for (const rule of rules) {
    const conditionsMet = evaluateConditions(rule.conditions, context);
    if (conditionsMet) {
      await executeActions(rule.actions, context);
      await logExecution(rule.id, context, 'success');
    }
  }
}
```

### 6.4 Failure Handling in Rules

Rule execution failures must not break the main operation:

```typescript
try {
  await executeActions(rule.actions, context);
  await logExecution(rule.id, context, 'success');
} catch (error) {
  await logExecution(rule.id, context, 'failed', error.message);
  await NotificationService.notifyAdmin('business_rule_failed', { rule_id: rule.id });
  // Continue — do not throw — main operation already succeeded
}
```

## 7. AI Integration Strategy

### 7.1 Provider Abstraction

AI calls are abstracted behind `AIProviderClient`. The concrete provider (Claude) is configured via environment variable:

```typescript
// lib/ai/provider.ts
export class AIProviderClient {
  async analyze(
    systemPrompt: string,
    userPrompt: string,
    context: Record<string, unknown>,
  ): Promise<AIOutput> {
    // Store request in ai_requests
    // Call Claude claude-sonnet-4-6 via Anthropic SDK
    // Retry on failure (max 3, exponential backoff)
    // Store response in ai_responses
    // Return structured output
  }
}
```

### 7.2 Prompt Context Injection

Every agent receives security-scoped context:

```typescript
const context = {
  company_id: user.company_id,
  site_ids: user.authorized_sites,
  user_role: user.roles,
  // Agent-specific records — only what is needed for this request
};
```

Context is never constructed from client-provided data. It is always derived from the authenticated session.

### 7.3 Structured Output Enforcement

Every agent prompt demands JSON output. The response is validated against a Zod schema before being stored:

```typescript
const output = await aiClient.analyze(systemPrompt, userPrompt, context);
const validated = agentOutputSchema.safeParse(output);
if (!validated.success) {
  // Log parse failure, return error to user
  // Do not store malformed AI output
}
```

### 7.4 OCR Strategy

Claude's vision capability (multimodal) handles document OCR in MVP:

```typescript
// Edge function: document-ocr
const response = await anthropic.messages.create({
  model: 'claude-sonnet-4-6',
  messages: [
    {
      role: 'user',
      content: [
        {
          type: 'image',
          source: { type: 'base64', data: pdfBase64, media_type: 'application/pdf' },
        },
        { type: 'text', text: 'Extract all text from this document. Return the raw text content.' },
      ],
    },
  ],
});
```

Extracted text is stored in `file_ai_metadata.extracted_text` and indexed for FTS.

## 8. Supabase Edge Functions Strategy

Edge Functions are used only for workloads that require server-side execution outside of Next.js API routes:

| Function                  | Trigger                       | Purpose                                 |
| ------------------------- | ----------------------------- | --------------------------------------- |
| `protocol-ai`             | HTTP from API route           | Run Protocol Agent against uploaded PDF |
| `regulatory-ai`           | HTTP from API route           | Run Regulatory Agent against document   |
| `analytics-summary`       | Cron (hourly)                 | Aggregate KPIs across all companies     |
| `document-ocr`            | HTTP from FileService         | Extract text via Claude vision          |
| `task-generator`          | HTTP from services            | Create task + send realtime event       |
| `notification-dispatcher` | HTTP from NotificationService | Email dispatch + in-app write           |

Edge Functions call the Supabase admin client (service role) only for operations that require elevated access (e.g., writing to a table the user doesn't directly own).

## 9. Testing Strategy

### 9.1 Test Pyramid

```
Unit Tests (Vitest)         → Service-layer logic, utility functions
Integration Tests (Vitest)  → API routes with test database, RLS policy tests
E2E Tests (Playwright)      → Critical user flows in a real browser
```

### 9.2 Critical Test Scenarios (Every Sprint)

These scenarios must pass before a sprint is considered done:

```
Security:
  - User from Company A cannot read Company B's data (RLS test)
  - User without permission X cannot call API that requires permission X
  - Non-admin cannot modify other users' data

Data Integrity:
  - Creating a subject generates the correct number of visits
  - Completing a visit creates exactly one chart (no duplicates)
  - Entering a chart in EDC records entered_by and entered_at

Business Rules:
  - Overdue charts are prioritized correctly
  - Document expiration triggers notification task
  - Subject early termination cancels future visits

AI:
  - AI response is never applied without requires_review = false or user approval
  - AI request is always logged in ai_requests
```

### 9.3 RLS Test Pattern

```typescript
// tests/integration/rls/company-isolation.test.ts
it('user from company A cannot read company B subjects', async () => {
  const companyA_client = createSupabaseClient({ user: companyA_user });
  const companyB_subject = await createTestSubject(companyB);

  const { data } = await companyA_client.from('subjects').select('*').eq('id', companyB_subject.id);

  expect(data).toHaveLength(0); // RLS must return 0 rows, not an error
});
```

## 10. Performance Strategy

### 10.1 Query Optimization Rules

```
Rule 1: No N+1 queries.
  - Use Supabase nested selects or JOIN queries.
  - Load tab data lazily (only when tab is clicked).

Rule 2: Paginate all lists.
  - Default page size: 25 rows.
  - Maximum page size: 100 rows.
  - Use cursor-based pagination for large tables (subjects, audit_logs).

Rule 3: Index all foreign keys and filter columns.
  - Every WHERE clause column that doesn't already have a PK must have an index.
  - Composite indexes for (company_id + site_id) pattern (most common filter).

Rule 4: Cache permission resolution.
  - User permissions are loaded once at login.
  - Stored in session context (JWT claims or server session).
  - Not re-queried on every API request.

Rule 5: KPI calculations are incremental.
  - Never recalculate all KPIs from scratch on every page load.
  - AnalyticsService.recalculate() updates only the affected KPI row.
  - Heavy aggregations run via cron Edge Function (hourly).
```

### 10.2 Realtime Scope Limits

Realtime subscriptions are expensive. Subscribe only to what changes frequently and affects the current user:

```
Subscribe:  tasks (assigned to current user)
Subscribe:  notifications (for current user)
Do NOT subscribe: subjects, visits, charts (use manual refresh or SWR)
```

### 10.3 Target Performance Metrics

```
API response time:        < 250ms (p95)
Page initial load:        < 2s (LCP)
Chart queue render:       < 1s (50 charts)
Subject profile load:     < 500ms
AI response:              < 30s (acceptable for protocol extraction)
```

## 11. Deployment Strategy

### 11.1 Environment Progression

```
Local Development:
  - Supabase local (supabase start)
  - Next.js dev server (pnpm dev)
  - .env.local with local Supabase credentials

Staging (auto-deploy from main branch):
  - Vercel Preview deployment
  - Supabase staging project
  - Migrations run automatically via GitHub Action

Production (manual promote from staging):
  - Vercel Production deployment
  - Supabase production project
  - Migrations require manual approval + backup verification
```

### 11.2 Secrets Management

```
NEXT_PUBLIC_SUPABASE_URL        → Vercel env vars (not secret, public)
NEXT_PUBLIC_SUPABASE_ANON_KEY   → Vercel env vars (not secret, public)
SUPABASE_SERVICE_ROLE_KEY       → Vercel env vars (SECRET, server-only)
ANTHROPIC_API_KEY               → Vercel env vars (SECRET, server-only, Edge Functions)
```

The `SUPABASE_SERVICE_ROLE_KEY` and `ANTHROPIC_API_KEY` must never appear in:

- Client-side bundles
- Browser network requests
- Server logs
- Error messages

### 11.3 CI/CD Pipeline

```yaml
# On every push:
1. pnpm lint         → ESLint + TypeScript check
2. pnpm test:unit    → Vitest unit tests
3. pnpm build        → Next.js production build

# On main branch merge:
4. pnpm test:integration → Integration tests against test DB
5. Vercel auto-deploy     → Staging environment
6. Supabase migrations    → Run pending migrations on staging

# On production release (manual):
7. Backup Supabase production DB
8. Run migrations on production
9. Deploy to Vercel production
10. Run smoke tests
```

## 14. Non-Negotiable Architecture Rules

These rules cannot be overridden by any sprint plan, schedule pressure, or user request:

```
1. RLS is mandatory on every business table. Never disable.
2. AI never writes production data without human approval.
3. Every critical action is audited. Audit logs are immutable.
4. Business logic lives in services. Never in React components.
5. Files are never public. Always signed URLs with expiry.
6. Company_id isolation is verified at service layer, not just RLS.
7. The service role key never appears in client code or browser requests.
8. Migrations only. Never manual schema changes in production.
9. TypeScript strict mode. No 'any'. Server-side validation required.
10. Business Rules fire on every state transition. No silent writes.
```

Violating any of these rules requires explicit written authorization from the project lead and must be documented in the decision log.
