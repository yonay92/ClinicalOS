## Sprint 9 — Business Rules Engine

**Goal:** Configurable rule builder and execution engine.

### Business Rule JSON Schema (GAP-BL-01 Resolution)

```typescript
// conditions schema
type RuleCondition = {
  field: string; // e.g., "chart.days_pending"
  operator: '>' | '<' | '>=' | '<=' | '==' | '!=' | 'in' | 'not_in';
  value: number | string | string[];
  logic?: 'AND' | 'OR'; // for combining multiple conditions
};

// actions schema
type RuleAction = {
  type:
    | 'create_task'
    | 'update_status'
    | 'send_notification'
    | 'create_chart'
    | 'calculate_kpi'
    | 'write_audit';
  payload: Record<string, unknown>;
};
```

### Services

```
BusinessRuleEngine
  - evaluate(trigger: string, context: Record<string, unknown>)
    → finds active rules matching trigger
    → evaluates conditions against context
    → executes actions
    → logs to rule_execution_logs

RuleEvaluator
  - evaluateCondition(condition: RuleCondition, context)  → boolean
  - executeAction(action: RuleAction, context)            → void
```

### API Routes

```
GET    /api/business-rules
POST   /api/business-rules
PATCH  /api/business-rules/:id
POST   /api/business-rules/:id/test
POST   /api/business-rules/:id/execute
```

### UI Pages

```
/app/(app)/settings/business-rules/page.tsx → Rule list
```

### Default Rules (seed data)

```
- chart_priority_critical:   chart.days_pending > 7 → priority = critical
- chart_priority_high:       chart.days_pending in [4..7] → priority = high
- doc_expiring_90:           doc.expiration_date within 90 days → create_renewal_task
- doc_expiring_30:           doc.expiration_date within 30 days → notify_regulatory
- visit_out_of_window:       visit.completion_date outside window → increase_priority + notify_crc
- subject_early_term:        subject.status = early_terminated → cancel_future_visits
- study_closeout:            study.status = closed → archive_binder + generate_closeout_tasks
```

### Definition of Done

- Admin can view, create, edit, disable rules via UI
- Engine evaluates correct rules on every trigger
- All rule executions logged in `rule_execution_logs`
- Default system rules run automatically
- Admin can test a rule against sample data before activating
- System failure during rule execution: logs failure, notifies admin

## Sprint 13 — Security Hardening

**Goal:** Complete security audit, RLS review, rate limiting, pen test checklist.

### Tasks

```
1. Review every RLS policy for correctness
2. Test cross-company isolation (automated test suite)
3. Test site isolation (automated test suite)
4. Implement API rate limiting (Vercel Edge + Upstash Redis)
5. Implement AI request rate limiting per user/day
6. Enable Supabase MFA enrollment (optional for users)
7. Audit all signed URL configurations
8. Verify SUPABASE_SERVICE_ROLE_KEY is never exposed in client code
9. Validate all API inputs with Zod schemas
10. Review error messages (no SQL/stack traces exposed)
11. Test file access authorization (signed URL requires valid session)
12. Security monitoring: track failed logins in audit_logs
```

### Tests

```
security: rls-isolation.test.ts (comprehensive)
security: api-authorization.test.ts
security: file-access.test.ts
```

### Definition of Done

- Zero cross-company data exposure (automated tests pass)
- All API endpoints require authentication
- Rate limiting active on auth and AI endpoints
- No service role key in any client-side code
- All inputs validated server-side

## Sprint Velocity Guidelines

Each sprint is designed to be completable in approximately **2-3 weeks** of focused development, given Claude Code assistance.

The sprint sequence must not be reordered. Dependencies between sprints are hard.

Sprints 1-5 are the critical path. Delays here cascade to all subsequent sprints.
