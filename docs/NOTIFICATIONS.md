# ClinicalOS — Notification Architecture Specification

Version: 1.0
Status: Production-Ready — Required before Sprint 1 (tables) / Sprint 8 (full implementation)

---

## 1. Purpose

This document defines the complete notification system for ClinicalOS. It resolves **GAP-DB-01** from GAP_ANALYSIS.md by specifying the `notifications` and `notification_preferences` tables, their RLS policies, the dispatch service, the delivery channels, the event type catalog, and the real-time subscription model.

---

## 2. Core Principles

1. Notifications are always company-scoped and user-scoped — no cross-company notification is ever created.
2. Notifications are generated by the Business Rule Engine or by explicit service calls — never directly from UI components.
3. Critical events always generate in-app notifications; email is controlled by user preferences.
4. Users control notification preferences per event type and per channel.
5. All notification dispatch is asynchronous — it never blocks the primary operation.
6. Notifications are delivered via in-app real-time and email. SMS is reserved for v1.5.

## 3. Database Schema

### 3.1 `notifications` table (new — add to migration 001)

```sql
CREATE TABLE notifications (
  id                uuid         PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id        uuid         NOT NULL REFERENCES companies(id),
  user_id           uuid         NOT NULL REFERENCES profiles(id),
  type              text         NOT NULL,   -- see §6 for event type catalog
  title             text         NOT NULL,
  body              text,
  related_module    text,        -- e.g., 'regulatory', 'charts', 'tasks'
  related_record_id uuid,        -- polymorphic FK to the triggering record
  related_record_type text,      -- e.g., 'regulatory_document', 'chart', 'task'
  priority          text         NOT NULL DEFAULT 'medium', -- critical | high | medium | low
  is_read           boolean      NOT NULL DEFAULT false,
  read_at           timestamptz,
  created_at        timestamptz  NOT NULL DEFAULT now()
);

-- Indexes
CREATE INDEX idx_notifications_user       ON notifications(user_id, created_at DESC);
CREATE INDEX idx_notifications_company    ON notifications(company_id);
CREATE INDEX idx_notifications_unread     ON notifications(user_id, is_read) WHERE is_read = false;
CREATE INDEX idx_notifications_module     ON notifications(related_module, related_record_id);
```

### 3.2 `notification_preferences` table (new — add to migration 001)

```sql
CREATE TABLE notification_preferences (
  id          uuid         PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id  uuid         NOT NULL REFERENCES companies(id),
  user_id     uuid         NOT NULL REFERENCES profiles(id),
  event_type  text         NOT NULL,  -- see §6 for event type catalog
  in_app      boolean      NOT NULL DEFAULT true,
  email       boolean      NOT NULL DEFAULT true,
  created_at  timestamptz  NOT NULL DEFAULT now(),
  updated_at  timestamptz  NOT NULL DEFAULT now(),
  CONSTRAINT uq_notification_prefs UNIQUE(user_id, event_type)
);

CREATE INDEX idx_notification_prefs_user ON notification_preferences(user_id);
```

### 3.3 RLS Policies

```sql
-- notifications: users see only their own notifications
ALTER TABLE notifications ENABLE ROW LEVEL SECURITY;

CREATE POLICY "notifications_select" ON notifications
  FOR SELECT USING (
    company_id = current_company_id()
    AND user_id = auth.uid()
  );

CREATE POLICY "notifications_update" ON notifications
  FOR UPDATE USING (
    company_id = current_company_id()
    AND user_id = auth.uid()
  )
  WITH CHECK (
    company_id = current_company_id()
    AND user_id = auth.uid()
  );

-- notifications are created server-side only (via service role in Edge Functions)
-- No INSERT or DELETE policy for authenticated users

-- notification_preferences: users manage only their own preferences
ALTER TABLE notification_preferences ENABLE ROW LEVEL SECURITY;

CREATE POLICY "notification_prefs_all" ON notification_preferences
  FOR ALL USING (
    company_id = current_company_id()
    AND user_id = auth.uid()
  )
  WITH CHECK (
    company_id = current_company_id()
    AND user_id = auth.uid()
  );
```

## 4. Notification Service

### 4.1 Service Interface

```typescript
// services/NotificationService.ts

export interface NotificationDispatchInput {
  type: NotificationEventType; // from the event catalog (§6)
  companyId: string;
  siteId?: string;
  recipientRole?: string; // dispatch to all users with this role at the site
  recipientUserId?: string; // OR dispatch to a specific user
  relatedModule?: string;
  relatedRecordId?: string;
  relatedRecordType?: string;
  customTitle?: string; // overrides the default title template
  customBody?: string; // overrides the default body template
  priority?: 'critical' | 'high' | 'medium' | 'low';
  context?: Record<string, string>; // template variable substitutions
}

export class NotificationService {
  async dispatch(input: NotificationDispatchInput): Promise<void>;
  async markAsRead(notificationId: string, userId: string): Promise<void>;
  async markAllAsRead(userId: string, companyId: string): Promise<void>;
  async getUnreadCount(userId: string, companyId: string): Promise<number>;
  async getUserNotifications(
    userId: string,
    companyId: string,
    limit?: number,
  ): Promise<Notification[]>;
  async getUserPreferences(userId: string, companyId: string): Promise<NotificationPreference[]>;
  async updatePreference(
    userId: string,
    eventType: string,
    channels: { in_app: boolean; email: boolean },
  ): Promise<void>;
}
```

### 4.2 Dispatch Logic

```typescript
async function dispatch(input: NotificationDispatchInput): Promise<void> {
  // 1. Resolve recipients
  const recipients = await resolveRecipients(input);
  if (recipients.length === 0) return;

  // 2. For each recipient, check preferences and dispatch
  for (const userId of recipients) {
    const prefs = await getPreferences(userId, input.type);

    if (prefs.in_app) {
      await createInAppNotification(userId, input);
    }

    if (prefs.email) {
      await queueEmailNotification(userId, input);
    }
  }
}

async function resolveRecipients(input: NotificationDispatchInput): Promise<string[]> {
  if (input.recipientUserId) return [input.recipientUserId];

  if (input.recipientRole && input.siteId) {
    // Return all active users with the given role at the given site
    return db.profiles
      .joinUserRoles()
      .joinUserSites()
      .where({
        company_id: input.companyId,
        site_id: input.siteId,
        role_key: input.recipientRole,
        status: 'active',
      })
      .select('id');
  }

  return [];
}

async function getPreferences(
  userId: string,
  eventType: string,
): Promise<{ in_app: boolean; email: boolean }> {
  const pref = await db.notification_preferences.findOne({
    user_id: userId,
    event_type: eventType,
  });
  // Fall back to defaults if no explicit preference set
  return pref ?? { in_app: true, email: true };
}
```

## 5. In-App Notification: Real-Time Delivery

In-app notifications use Supabase Realtime. When a `notifications` record is inserted, the subscribing client receives the event immediately.

### 5.1 Channel Naming Convention (resolves GAP-REQ-04)

```
company:{company_id}:user:{user_id}:notifications
```

Each authenticated user subscribes to their own channel on login. The channel is scoped by both `company_id` and `user_id` to prevent cross-user data leakage.

### 5.2 Client Subscription

```typescript
// hooks/useNotifications.ts

export function useNotifications() {
  const { user, company } = useAuth();

  useEffect(() => {
    const channel = supabase
      .channel(`company:${company.id}:user:${user.id}:notifications`)
      .on(
        'postgres_changes',
        {
          event: 'INSERT',
          schema: 'public',
          table: 'notifications',
          filter: `user_id=eq.${user.id}`,
        },
        (payload) => {
          addNotificationToState(payload.new as Notification);
          showToast(payload.new as Notification);
        },
      )
      .subscribe();

    return () => {
      supabase.removeChannel(channel);
    };
  }, [user.id, company.id]);
}
```

### 5.3 Notification Bell UI

- The notification bell in the TopBar shows unread count (loaded on mount, updated via Realtime).
- Clicking the bell opens a dropdown showing the 20 most recent notifications.
- Notifications are grouped: Critical (red) → High (orange) → Medium (blue) → Low (grey).
- Each notification links to the related record via `related_module` + `related_record_id`.
- "Mark all as read" is available in the dropdown header.

## 7. Email Delivery

Email notifications are dispatched asynchronously via Supabase Edge Function `notification-dispatcher`.

### 7.1 Queue Pattern

The `NotificationService` writes an email job to a lightweight queue table rather than calling the Edge Function synchronously:

```sql
-- notification_email_queue (new — add to migration 001)
CREATE TABLE notification_email_queue (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id      uuid        NOT NULL REFERENCES companies(id),
  user_id         uuid        NOT NULL REFERENCES profiles(id),
  notification_id uuid        NOT NULL REFERENCES notifications(id),
  email           text        NOT NULL,
  subject         text        NOT NULL,
  html_body       text        NOT NULL,
  status          text        NOT NULL DEFAULT 'pending', -- pending | sent | failed
  attempts        integer     NOT NULL DEFAULT 0,
  last_attempted  timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_email_queue_pending ON notification_email_queue(status, created_at) WHERE status = 'pending';
```

### 7.2 Edge Function: `notification-dispatcher`

Runs on a 1-minute cron. Processes up to 50 pending emails per invocation. Uses Supabase's built-in SMTP (or Resend as an alternative) for transactional delivery.

```typescript
// supabase/functions/notification-dispatcher/index.ts
// Picks up to 50 pending records from notification_email_queue
// Sends via Resend (RESEND_API_KEY env var)
// Updates status to 'sent' or 'failed'
// Failed records are retried up to 3 times before being marked permanently failed
```

### 7.3 Email Provider

- **MVP:** Supabase SMTP (built-in) for simplicity.
- **v1.5:** Resend (`RESEND_API_KEY`) for better deliverability, template management, and delivery tracking.

### 7.4 Grouping

Medium and low priority email notifications are batched into a daily digest (sent at 08:00 in the company's timezone). Critical and high priority emails are sent immediately.

| Priority | Email Timing |
| -------- | ------------ |
| critical | Immediately  |
| high     | Immediately  |
| medium   | Daily digest |
| low      | Daily digest |

## 10. Security

- Notification records are user-owned — RLS ensures a user can only read their own notifications.
- Notifications are created server-side with the service role key (inside Edge Functions or server-only services). Authenticated users cannot INSERT notifications directly.
- The email queue is not exposed to any API endpoint — it is processed exclusively by the Edge Function.
- Notification content must never include raw PHI. Reference the related record by ID — the UI loads the actual data separately with proper permission checks.
- Cross-site notifications are not permitted: a notification for a site-specific event may only be sent to users authorized for that site.
