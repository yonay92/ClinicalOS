### GAP-DB-03: `files` and `file_folders` lack `site_id`

**Impact:** MEDIUM

**Source:** SECURITY.md Â§4 states site-specific tables must include `site_id`. `files` and `file_folders` only have `company_id`. Files are accessed by site-specific modules (regulatory, subjects). Without `site_id` on `files`, site-level RLS cannot be applied directly.

**Recommendation:**

- The RLS policy on file access must check: the requesting user has access to the linked record's site.
- Document this clearly as the intended design â€” it is indirect but correct for the polymorphic file model.

**Resolution:** Add explicit documentation to RLS policies; ensure `file_links` carries correct records with site-validated foreign keys.

### GAP-DB-04: `regulatory_documents` has no `binder_id` FK

**Impact:** MEDIUM

## 2. Underspecified Business Logic

### GAP-BL-01: Business Rule Engine execution model undefined

**Impact:** CRITICAL

**Source:** DATABASE_Part_05 defines `business_rules` with `conditions jsonb` and `actions jsonb`. BUSINESS_RULES_01 defines rule types (document_expiration, visit_generation, chart_priority, task_creation, notification, ai_workflow). But HOW the engine evaluates JSONB conditions at runtime is not specified.

**Questions to resolve:**

- Is the rule engine server-side (Node.js) or database-side (PL/pgSQL triggers)?
- What is the JSON schema for `conditions`? Example: `{"field": "chart.days_pending", "op": ">", "value": 7}`?
- What is the JSON schema for `actions`? Example: `{"type": "create_task", "title": "Chart overdue", "priority": "critical"}`?
- Who invokes the engine? Is it called from each service after every write?

**Recommendation:** Define a concrete JSON schema for conditions and actions. Implement the engine in TypeScript in `BusinessRuleEngine.ts`. Call it from every service after each write. Log every evaluation in `rule_execution_logs`.

---

### GAP-BL-02: `analytics_kpis` update mechanism undefined

**Impact:** HIGH

**Source:** BUSINESS_RULES_09 says "KPIs are recalculated incrementally" with triggers from 5 event types. SUPABASE_SETUP.md mentions "incremental KPI calculations." No trigger mechanism is specified (DB triggers, Edge Functions, cron jobs).

**Recommendation:**

- Use service-layer calls: every service that affects a KPI calls `AnalyticsService.recalculate(kpi_name, context)` after completing its write.
- No DB triggers â€” they create invisible side effects that are hard to debug.
- For heavy aggregations, use a Supabase Edge Function triggered on-demand or via cron.

### GAP-BL-03: Visit rescheduling cascade logic underspecified

**Impact:** MEDIUM

**Source:** API.md Â§12 defines three reschedule options: `only_this_visit`, `this_and_future_visits`, `keep_original_future_schedule`. The business logic for `this_and_future_visits` needs precise definition.

**Questions:**

- When rescheduling all future visits, do the offsets recalculate from the new visit date or from the baseline date?
- If future visits have already been completed, are they affected?

**Recommendation:** Visits recalculate offsets from the `subjects.baseline_date`. Already-completed visits are never rescheduled. Only `scheduled` and `confirmed` future visits are affected.

## 5. Overlapping/Redundant Concepts

### GAP-DUP-01: `subject_timeline` vs `subject_status_history`

**Overlap:** Both tables record subject events. `subject_status_history` captures status changes specifically. `subject_timeline` captures all events with a `related_record_type` link.

**Resolution:** Both tables are intentionally kept. When a status changes:

1. Write to `subject_status_history` (structured status record)
2. Write to `subject_timeline` (narrative event for display)
3. Write to `audit_logs` (immutable audit record)
   All three serve distinct purposes. Services must write to all three on status change.

---

### GAP-DUP-02: `document_versions` vs `file_versions`

**Overlap:** Both `document_versions` (in `regulatory_documents`) and `file_versions` (in `files`) track versioning.

**Resolution:**

- `file_versions` tracks the binary file versions (storage path, checksum).
- `document_versions` tracks regulatory document metadata versions (effective date, version string, status).
- A new regulatory document version creates both a new `file_versions` record and a new `document_versions` record.
- These are complementary, not duplicative.

---

### GAP-DUP-03: `charts` priority vs `chart_metrics` data

**Overlap:** `charts.priority` (computed) and `chart_metrics.overdue_days` are related. Priority is derived from the same metrics.

**Resolution:** `charts.priority` is the cached computed value (critical/high/medium/low). `chart_metrics` stores the raw inputs. The BusinessRuleEngine reads `chart_metrics` to set `charts.priority`. Priority must be recalculated when `chart_metrics` changes.

## 7. Scalability Concerns

### GAP-SCALE-01: `audit_logs` table will grow unboundedly

**Impact:** MEDIUM

**Source:** Every critical action writes an immutable audit log. No retention or archival policy is defined.

**Recommendation:**

- Add a `company_id + created_at` composite index (already listed in GAP-IDX section).
- Implement a data retention policy configurable per company (default: 7 years for clinical data).
- Archive old audit logs to cold storage (Supabase Storage) after retention period.
- Do not hard-delete audit logs.

---

### GAP-SCALE-02: `rule_execution_logs` volume under heavy business rule activity

**Impact:** MEDIUM

**Source:** Every rule evaluation is logged. High-frequency events (document expirations checked daily for thousands of documents) will create massive log volume.

**Recommendation:**

- Add `result` column filtering (only log `success` with changes and all `failure` results).
- Purge `rule_execution_logs` after 90 days (unlike audit_logs which are permanent).
- Consider batching rule evaluations for expiration checks.

---

### GAP-SCALE-03: `analytics_kpis` table design is insufficient for complex queries

**Impact:** MEDIUM

**Source:** `analytics_kpis` stores KPI rows with `kpi_name`, `kpi_value`, `period_type`. For a multi-study, multi-site organization, the number of KPI records grows multiplicatively.

**Recommendation:** Consider a JSONB column per KPI row for multi-dimensional data, or use materialized views for complex analytics. For MVP, the current design is acceptable.

## 8. Performance Concerns

### GAP-PERF-01: N+1 query risk in Subject Profile

**Impact:** HIGH

**Source:** `GET /api/subjects/:id` returns "overview, visits, charts, timeline, notes, documents, history." A naive implementation would make separate queries for each tab.

**Recommendation:** Use Supabase's `select()` with nested relations to fetch all related data in one query per section. Implement lazy loading per tab (fetch tab data only when the tab is opened).

---

### GAP-PERF-02: Chart queue ordering requires complex sorting

**Impact:** MEDIUM

**Source:** The chart queue must sort by: (1) critical overdue, (2) out of window, (3) sponsor-related, (4) remaining. This cannot be done with a simple `ORDER BY` without calculated columns.

**Recommendation:** Implement `calculate_chart_priority()` as a SQL function (already in SUPABASE_SETUP spec). Add a `priority_score numeric` column to `charts` that is computed by the function and updated by the BusinessRuleEngine. Sort queue by `priority_score DESC, created_at ASC`.

## 9. Missing Requirements

### GAP-REQ-01: No specified behavior for concurrent visit completions

**Source:** If two CRCs simultaneously complete the same visit, the system could create two charts.

**Recommendation:** Add a `UNIQUE(visit_id)` constraint on `charts.visit_id`. The second chart creation will fail gracefully with a `DUPLICATE_RECORD` error.

---

### GAP-REQ-02: No subject number uniqueness scope defined

**Source:** `subject_number` is defined as `text not null` without a unique constraint. Clinical trials require unique subject numbers per study.

**Recommendation:** Add: `UNIQUE(study_id, site_id, subject_number)` or `UNIQUE(study_id, subject_number)` depending on whether subject numbers are study-wide or site-specific. Verify with domain experts. Default recommendation: `UNIQUE(study_id, subject_number)`.

---

### GAP-REQ-03: No defined behavior when visit template is not yet approved

**Source:** BUSINESS_RULES_03 says subject creation generates a visit schedule. But what if no approved visit template exists for the study?

**Recommendation:** Block subject creation if no approved visit template exists. Display a clear error: "This study does not have an approved visit template. Please approve a visit template before creating subjects." Add this as a validation rule in `SubjectService`.

---

### GAP-REQ-04: Realtime channel scope undefined

**Source:** SUPABASE_SETUP.md Â§8 enables Realtime for tasks, notifications, charts, calendar, AI suggestions. No channel naming convention or subscription scope (company-wide vs. site-specific) is defined.

**Recommendation:**

- Channel format: `company:{company_id}:tasks`, `company:{company_id}:site:{site_id}:notifications`
- Subscribe on the client using the authenticated user's company and authorized sites.
- Only subscribe to tables that actually need real-time updates (tasks, notifications).
